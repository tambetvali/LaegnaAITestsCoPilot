# Moon Notebook

https://jupyter.org/ is the well-known Jupyter Notebook, also used in Google CoLab and other experiment and toolchain systems.

As a programmer, strong advocate of simple formats such as UTF-8 and Markdown for text editing (or ASCII in the past, rather than RTF or painting text to paint images), I see the Jupyter format complex and hard to read.

Jupyter is Json, which is known for:
- Being extremely simple to parse for programmers, as freeform data structure; it's an automatic analogous to Markdown, which has freeform data for users, in form of chapters and paragraphs, not variables and lists.

I do not support using Json types in our tradeoff:
- Json for AI card input
- Json for Notebooks
- Json for even containing our threads and blogs

Json *is* kind of human-readable, as it's also ASCII or UTF-8 based, rather than binary format which also uses underlying number-encoding in addition overlying to letter-encoding. I am sure we can produce JSON, for example, as last input for AI fine-tuner, but I do not think it's better than specialized, strict-standard filetype such as Anki, and the Markdown files which let the cards to be generated. File typing very strongly influences our process and it's understandability and ability to cooperate, altough it's psychological influence is hard to explain: rather we see programs, tools and data than bites and bits and floats and paddings inside, in each case where we choose a tool.

## My idea

Markdown, essentially *already looks like a Notebook* and this is what we should utilize:
- It supports code blocks
- It supports meaningful text in between of code blocks

Moon notebook is thus, a Markdown format, which would apply interpreters or other code-runners to work on our document, and it does so:
- Assigns one console block after each well-known code block.
- Sees the code blocks (optional) as parts of the same program.
- Perhaps, information might still be fed to code blocks before.

It might garbage you a little with an autogenerated cache: past outputs, whether the blocks have been ran or are part of the flow; internals could be stored here.

It might have also this view: an automation, where *opening your file^ itself means that you won't edit it, but it will be displayed, and all the code blocks executed in order unless the language is not known by it's interpreter and output format.

You can extend the code blocks to read variables, GET/POST or input fields you can put into Markdown: probably, a single extension would connect to browser, it's server or current session.

## How does execution work?

- After each code block, an output block is inserted  
- A cache folder stores large outputs  
- Opening the file in ‚Äúrun mode‚Äù executes everything
